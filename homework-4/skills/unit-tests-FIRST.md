# 🧪 Skill: Unit Tests FIRST Principles

## Overview

This skill defines the **FIRST** principles for writing high-quality unit tests. All unit tests generated by the Unit Test Generator agent must adhere to these principles.

---

## FIRST Principles

### **F** — Fast ⚡

Tests must execute quickly.

| Criteria | Description |
|----------|-------------|
| **Target** | Each test should complete in < 100ms |
| **Suite** | Full test suite should run in < 10 seconds |
| **Why** | Fast tests encourage frequent running |

**Guidelines:**
- Avoid I/O operations (file, network, database) in unit tests
- Use mocks/stubs for external dependencies
- No sleep/wait/delay calls
- Test small units of code

**Anti-patterns to avoid:**
```javascript
// ❌ Slow: Real network call
const response = await fetch(API_URL);

// ❌ Slow: Artificial delay
await new Promise(r => setTimeout(r, 1000));

// ❌ Slow: Real database
const user = await database.findUser(id);
```

**Correct approach:**
```javascript
// ✅ Fast: Mocked dependency
const mockDb = { findUser: jest.fn().mockResolvedValue(mockUser) };
```

---

### **I** — Independent 🔀

Tests must not depend on each other.

| Criteria | Description |
|----------|-------------|
| **Isolation** | Each test runs in isolation |
| **Order** | Tests can run in any order |
| **State** | No shared mutable state between tests |

**Guidelines:**
- Each test sets up its own data
- Each test cleans up after itself
- No test relies on another test's output
- Use `beforeEach`/`afterEach` for setup/teardown

**Anti-patterns to avoid:**
```javascript
// ❌ Dependent: Shared state
let user;
test('create user', () => { user = createUser(); });
test('update user', () => { updateUser(user); });  // Depends on previous test!
```

**Correct approach:**
```javascript
// ✅ Independent: Each test has own setup
test('create user', () => {
  const user = createUser();
  expect(user).toBeDefined();
});

test('update user', () => {
  const user = createUser();  // Own setup
  const updated = updateUser(user);
  expect(updated).toBeDefined();
});
```

---

### **R** — Repeatable 🔁

Tests must produce the same result every time.

| Criteria | Description |
|----------|-------------|
| **Deterministic** | Same input → Same output |
| **Environment** | Works on any machine |
| **Time** | Not affected by current time/date |

**Guidelines:**
- No random values without seeding
- Mock current time when needed
- No environment-specific paths
- No reliance on external services

**Anti-patterns to avoid:**
```javascript
// ❌ Non-repeatable: Random
const id = Math.random();

// ❌ Non-repeatable: Current time
const isExpired = date < new Date();

// ❌ Non-repeatable: External service
const data = await externalAPI.getData();
```

**Correct approach:**
```javascript
// ✅ Repeatable: Fixed values
const id = 'test-id-123';

// ✅ Repeatable: Mocked time
jest.useFakeTimers().setSystemTime(new Date('2024-01-01'));

// ✅ Repeatable: Mocked external
const mockAPI = { getData: jest.fn().mockResolvedValue(testData) };
```

---

### **S** — Self-Validating ✅

Tests must have clear pass/fail outcome.

| Criteria | Description |
|----------|-------------|
| **Assertion** | Every test has assertions |
| **Clear** | Pass/fail is unambiguous |
| **Automated** | No manual verification needed |

**Guidelines:**
- Use explicit assertions (expect, assert)
- One logical assertion per test (can have multiple expect calls)
- Clear failure messages
- No console.log for verification

**Anti-patterns to avoid:**
```javascript
// ❌ No assertion
test('get user', () => {
  const user = getUser(123);
  console.log(user);  // Manual check required!
});

// ❌ Unclear assertion
test('user exists', () => {
  const user = getUser(123);
  expect(user).toBeTruthy();  // Too vague!
});
```

**Correct approach:**
```javascript
// ✅ Self-validating: Clear assertions
test('getUser returns user object with correct id', () => {
  const user = getUser(123);
  expect(user).toBeDefined();
  expect(user.id).toBe(123);
  expect(user.name).toBe('Alice');
});
```

---

### **T** — Timely ⏰

Tests must be written close to the code change.

| Criteria | Description |
|----------|-------------|
| **When** | Tests written with or before code changes |
| **Coverage** | Tests cover the specific changes made |
| **Relevant** | Tests are for current functionality |

**Guidelines:**
- Write tests for changed/added code immediately
- Don't defer test writing
- Tests should cover the fix specifically
- Update tests when code changes

**For Bug Fixes:**
- Write test that would have caught the bug
- Test covers the specific fix applied
- Test verifies correct behavior post-fix

**Correct approach:**
```javascript
// ✅ Timely: Test for the specific bug fix
// Bug: getUserById returned 404 due to string/number comparison
test('getUserById handles string ID parameter correctly', () => {
  const user = getUserById('123');  // String ID like from URL params
  expect(user.id).toBe(123);        // Returns user with numeric ID
});
```

## FIRST Compliance Checklist

Use this checklist to verify tests meet FIRST principles:

| Principle | Check | ✓ |
|-----------|-------|---|
| **Fast** | Test executes in < 100ms | ☐ |
| **Fast** | No real I/O operations | ☐ |
| **Fast** | External deps are mocked | ☐ |
| **Independent** | No shared mutable state | ☐ |
| **Independent** | Can run in any order | ☐ |
| **Independent** | Has own setup/teardown | ☐ |
| **Repeatable** | No random values | ☐ |
| **Repeatable** | Time is mocked if needed | ☐ |
| **Repeatable** | Works on any machine | ☐ |
| **Self-Validating** | Has explicit assertions | ☐ |
| **Self-Validating** | Clear pass/fail | ☐ |
| **Self-Validating** | No manual verification | ☐ |
| **Timely** | Covers changed code | ☐ |
| **Timely** | Written with the fix | ☐ |

## Test Quality Levels

| Level | FIRST Compliance | Description |
|-------|------------------|-------------|
| ⭐⭐⭐⭐⭐ | 100% | All FIRST principles fully satisfied |
| ⭐⭐⭐⭐ | 80-99% | Minor deviations, acceptable |
| ⭐⭐⭐ | 60-79% | Some principles not met |
| ⭐⭐ | 40-59% | Significant issues |
| ⭐ | < 40% | Tests need rewrite |
